Expanding Rich Media Ads and Iframes


BACKGROUND


AdMeld employs iframes in its ad tags (unless the publisher itself already uses them on its site(s)) to provide a layer of protection for publishers from issues that may arise from ad networks.  If an ad network is delayed in responding or there are other technical issues with the ad, by serving it through an iframe the rest of the page can load properly, preserving the end-user experience.  The content of the page also loads faster, as it does not have to wait for ads to complete loading.


Iframes however, do present an issue for expanding rich media ads, since iframes themselves are effectively self-contained boxes on a web page, preventing interaction between what is in the iframe and the web page it sits on.  In order to expand out of the iframe, the rich media vendor needs something else outside the iframe to latch on to, essentially, so they can break out of the box and render the expanding portion of the ad.  The hook for these expanding ads is a file the rich media vendors provide (a “pub-side file”) that needs to be placed on each site’s servers where the ads will run.



IMPLEMENTATION


For each of the rich media vendors (listed below), a single file needs to be installed on each domain and sub-domain on which the ads can run.


For example, if your site is www.publisherx.com but you also have ads on news.publisherx.com and video.publisherx.com, you need to install the respective files on each of these (so 12 total files would need to be installed: 3 domains x 4 vendors’ files).


The following provides all the information necessary to get the files installed for each rich media vendor.


POINTROLL

FILE NAME: PointRollAds.htm (case-sensitive)

LOCATION: Place on root directory for each domain

EXAMPLE(S): http://www.nbc.com/PointRollAds.htm

MORE INFO: http://www.pointroll.com/iframedelivery.aspx


DART MOTIF

FILE NAME: DARTIframe.html

LOCATION: Place in doubleclick folder on root directory for each domain

EXAMPLE(S): http://www.webmd.com/doubleclick/DARTIframe.html

MORE INFO: http://www.google.com/support/richmedia/bin/answer.py?answer=117857


EYEBLASTER

FILE NAME: addineyeV2.html

LOCATION: Place in eyeblaster folder on root directory for each domain

EXAMPLE(S): http://www.webmd.com/eyeblaster/addineyeV2.html

MORE INFO: http://www.eyeblasterwiz.com/onlinehelp/wwhelp/wwhimpl/js/html/wwhelp.htm?href=Expandable_Banner.htm


EYEWONDER

FILE NAME: interim.html

LOCATION: Place in eyewonder folder on root directory for each domain

EXAMPLE(S): http://www.nbc.com/eyewonder/interim.html

MORE INFO: contact Eyewonder at support@eyewonder.com


VIDEOEGG

FILE NAME: ad.html

LOCATION: Place in videoegg folder on root directory for each domain

EXAMPLE(S): http://www.nbc.com/videoegg/ad.html

MORE INFO:  http://developer.videoegg.com/mediawiki/index.php/VideoEgg_Ad_Platform_Integration_Guide_-_Website#Step_8._Upload_Iframe_Busting_File


EYENGAGE (TRAFFIC MARKETPLACE PROPRIETARY UNIT)

FILE NAME: eyengage.html

LOCATION: Please in trafficmp folder on root directory for each domain

EXAMPLE(S): http://www.nbc.com/trafficmp/eyengage.html

MORE INFO: http://www.eyengage.com/index.html



IMPORTANT POINTS


  1. 1) Ensure each file is named exactly as it appears here, including case (e.g. PointRollAds.htm, not pointrollads.htm)
  2. 2) Each rich media file needs to sit by itself, in its own location.  There should be no other code on the page, including your metrics trackers like Omniture, Hitbox, Google Analytics.  It should not be pasted into your CMS system, as that will usually include a lot of other javascript and HTML
  3. 3) Each domain and sub-domain where the ads could appear needs to have its own version of the files installed.  For example, news.yahoo.com and www.yahoo.com and movies.yahoo.com would each need their own versions of each set of pub-side files in place.
  4. 4) By default the rich media vendors will assume the file sits in the above locations (e.g. in the /eyewonder/ folder on root directory).  In most cases there are workarounds if the files cannot be named and placed as desired, but in general they present additional coordination and so should be avoided if possible.
  5. 5) IMPACT on site metrics data: if you do not remove the site tracking pixels from the pubside files’ pages, or you do not have them in place properly (which often results in additional hits against whichever page a 404 gets redirected to) you will see activity with tag.admeld.com listed as the referring URL.  These data points should be removed.  AdMeld shows up as the referring URL since our iframe calls the network, which calls the rich media vendor, who attempts to call your pubside file location (whether it is in place properly or not).  Putting the files in place properly and without any data tracking pixels on those pages will ensure you have no data anomalies or records of AdMeld as a top referring URL.
